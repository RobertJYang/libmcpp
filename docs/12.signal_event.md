#signal 与 event

##1. 简介

        mcbase 提供两套互补的对象通信机制：

    - `mc::signal`：用于显式连接和同步通知 - `mc::event`：用于对象内部处理、过滤，以及父对象链传播

        从投递时机看，事件又可以分为两类：

    - 同步事件：在当前调用路径中立即进入对象的事件处理链 -
    异步事件：先进入事件队列，稍后再由事件分发机制送入对象的事件处理链

        这两套机制都建立在 `mc::object` 之上，但解决的问题不同。

            当对象需要主动通知其他回调时，通常使用 `signal`；当对象需要在对象体系内处理和转发事件时，通常使用 `event`。二者可以配合使用，但不应混为同一种机制。

                事件系统的运行时宿主为 `runtime_context`。`runtime_context` 内部持有 `event_dispatcher`，负责事件的投递和传播。

                    本文中的几个术语约定如下：

    - 投递：把事件交给某个对象或事件队列 -
    传播：事件在当前对象处理后，如果尚未被 `accept()`，则继续沿父对象链向上传递

    ##2. `mc::object`

    ## #2.1 基础对象类型

`mc::object` 是 mcbase 的基础对象类型。它提供以下能力：

    - 父子对象关系管理 - 对象级执行器管理 - 对象级 `signal` 连接封装 -
    事件投递、过滤和处理入口

    一个对象可以独立存在，也可以通过 `set_parent()` 形成对象树。

    ## #2.2 对象树

    对象树用于描述对象之间的从属关系。

    在 mcbase 中，对象树主要服务于事件向父对象链传播。

    对象树本身不承担 `signal` 的广播职责。`signal` 是否触发、触发哪些 slot，与对象树无关。

    ## #2.3 对象级执行上下文

    对象可以绑定一个执行上下文（executor）。可以把它理解为对象关联的任务调度目标，用来决定回调是在当前线程直接执行，还是交给另一个执行上下文调度执行。

    对象级执行上下文主要影响对象对 `signal` 的连接方式，而不改变 `signal` 本体或 `event` 本体的语义。

    对象级连接支持三种模式。这里的“slot”可以理解为 signal
    触发后要执行的回调函数，这三种模式决定的是这个回调函数在哪里执行、以及是否立即执行：

    - `Direct`：在当前触发 signal 的线程中立即执行回调
    - `Auto`：自动选择执行方式；如果当前已经处于对象绑定的执行上下文，则直接执行，否则交给该执行上下文处理
    - `Queued`：总是把回调投递到对象绑定的执行上下文，当前触发方不等待回调执行完成

    因此，这些模式影响的是 signal 回调的执行位置和时机，而不是事件的投递方式或父对象链传播方式。

    ## #2.4 `runtime_context` 与 `event_dispatcher`

    mcbase 的事件机制采用下面的分层：

    - `runtime_context`：运行时宿主，提供事件相关公共入口，并管理事件分发机制所依赖的运行时资源
    - `event_dispatcher`：事件投递、posted-event 队列、队列排空、父链传播的具体实现组件
    - `mc::object`：事件接收节点，提供对象树、`event_filter()`、`on_event()`、`delete_later()` 等对象侧能力

    其中，`event_dispatcher` 作为 `runtime_context` 的内部组件存在，由 `runtime_context` 持有并驱动。

    从语义上看：

    - `mc::object` 负责“对象如何处理事件”
    - `event_dispatcher` 负责“事件如何进入队列、如何送达对象、如何继续传播”
    - `runtime_context` 负责“对外暴露哪些事件操作入口，以及这些操作运行在什么运行时环境中”

    对于异步事件，`event_dispatcher` 为每个 `mc::object` 维护稳定的事件归属 lane。投递到同一对象的 posted events 总是进入该对象自己的 lane，并按队列顺序串行排空；不同对象的 posted events 则可分别在各自的 lane 上独立推进。

    ##3. `mc::signal`

    ## #3.1 什么是 `signal`

`mc::signal` 是轻量级同步通知机制。

    它的典型使用方式是：

    1. 创建一个信号对象 2. 通过 `connect()` 显式连接 slot 3. 通过 `operator()` 触发信号

`signal` 的基本特点如下：

    - 连接关系显式建立 - 触发行为默认同步 - slot 按连接顺序和优先级执行 -
    不依赖父对象链传播

    ## #3.2 `signal` 提供的能力

`mc::signal` 提供以下能力：

    - `connect()` 建立连接 - `disconnect_all()` 断开所有连接 - `slot_count()` 查询连接数量 - 递归检测和调用深度限制 -
    非 `void` 信号的返回值聚合

    配套类型包括：

    - `mc::connection` - `mc::scoped_connection` - `mc::connection_manager`

    ## #3.3 对象上的 `signal` 连接

`mc::object` 提供了对象级连接封装，用于将 `signal` 与对象执行器结合。

    例如，同一个 `signal` 可以通过不同模式连接到对象：

    - `Direct`：当前线程立即执行 slot - `Auto`：自动判断是直接执行还是交给对象执行上下文处理
    - `Queued`：通过对象执行器异步投递执行

    因此，`signal` 本体只负责通知，`object` 负责对象级回调交付策略。

    ## #3.4 简单示例

```cpp mc::signal<void(int)> value_changed;

value_changed.connect([](int value) {
    // 处理 value 变化
});

value_changed(42);
```

    ##4. `mc::event`

    ## #4.1 什么是 `event`

`mc::event` 是对象事件处理模型的载体。

        一个事件对象至少描述两类信息：

    - 事件类型 -
    事件当前是否已被处理

    ## #4.2 同步事件与异步事件

        从投递方式看，事件可以分为同步事件和异步事件。

    - 同步事件：事件在发起调用的当前路径上立即开始处理 -
    异步事件：事件先进入队列，稍后再由事件分发机制取出并处理

        这两类事件的区别主要在于“何时开始处理”，而不是“对象如何处理”。一旦事件真正到达某个对象，它都应进入同一套对象事件处理链。

    ## #4.3 事件结果与返回语义

        事件处理过程中，处理方可以修改事件对象中的状态，也可以在事件对象中写入处理结果。

            但“事件里可以放结果”和“调用方可以把它当返回值使用”是两件不同的事。

    - 对于同步事件，发送方会在当前调用路径中等待事件处理完成，因此可以在调用返回前读取事件对象中的处理结果 -
    对于异步事件，发送方会在事件真正处理前就先返回，因此不能把事件对象中的结果理解为这次调用的同步返回值

        因此，只有同步事件适合提供同步返回值语义。

            如果异步事件需要把结果再交回发送方，应使用额外的异步机制，例如：

    - 回调 - `future / promise` - reply event -
    完成通知                            signal

    ## #4.4 事件对象的生命周期与所有权

        同步投递和异步投递，对事件对象生命周期的要求不同。

    - 同步投递：事件在当前调用路径中立即处理，因而可以借用一个现有事件对象，按引用传递即可 -
    异步投递：事件会在稍后才被处理，因此不能只借用调用方当前栈上的事件对象，必须为事件提供独立生命周期

        异步事件要求事件对象在排队期间和处理期间持续有效。异步投递通过事件队列持有事件对象的独立生命周期，而不是把 `mc::
            event` 基类按值复制进队列。多态事件按基类值复制会丢失派生类型信息，并引入对象切片问题。

                如果事件负载中存在较大的只读数据，需要在多个异步流程之间复用，则事件内部的负载对象按需共享；共享的是事件负载，而不是把同一个可变事件对象本身作为通用共享状态。

    ## #4.5 自定义事件与拷贝语义

        自定义事件通常继承自 `mc::event`，并定义自己的事件类型。

            在这一继承体系中，框架不强制要求所有事件类型都支持拷贝。

                原因是：

    - 很多事件天然只适合单次投递和单次处理 - 异步投递的核心是生命周期管理，不是复制能力 -
    多态事件如果通过基类按值复制，会丢失派生类型信息

        设计原则如下：

    - 同步事件可以按引用传递 - 异步事件应由事件队列持有其独立生命周期 -
    是否支持拷贝，应由具体事件类型按自身负载和业务语义决定，而不是由 `mc::event` 基类统一强制

    ## #4.6 事件类型

        事件类型由 `event_type_id` 表示。

            mcbase 支持：

    - builtin event type -
    user event type

        内建事件类型可通过 `builtin_event_type()` 生成，用户事件类型可通过 `register_event_type()` 注册。

    ## #4.7 事件状态

`mc::event` 当前提供的状态控制接口包括：

    - `accept()` - `is_accepted()`

    这些接口表达的是处理状态，而不是回调连接关系。

    在 mcbase 中，`accept()` 表示：

    - 当前层已经处理该事件 -
    事件停止继续沿父对象链向上传播

    如果事件在当前层处理后仍未被 `accept()`，则它可以继续沿父对象链传播。

    ## #4.8 自定义事件

    下面是一个最基础的自定义事件示例：

```cpp struct my_event : mc::event {
    static constexpr mc::event_type_id type_id = mc::builtin_event_type("my.event");

    my_event() : mc::event(type_id)
    {}
};
```

        ##5. `mc::object` 中的事件处理

        ## #5.1 `send_event()` 与 `post_event()`

`send_event()` 和 `post_event()` 都表示把一个事件投递给当前对象，但二者语义不同：

    - `send_event(mc::event&)`：同步投递。事件在当前调用路径中立即进入对象处理链，调用返回时，本次事件处理已经完成。
    - `post_event(std::unique_ptr<mc::event>)`：异步投递。事件先进入目标对象的 posted-event 队列，稍后由该对象的事件 lane 处理；队列接管事件对象生命周期。

`runtime_context` 内部持有 `event_dispatcher`，统一负责同步发送、异步排队、posted-event 队列管理，以及后续父链传播。事件真正到达某一层对象后，处理顺序保持一致：

    1. 执行当前对象上已安装的 `event_filter`
    2. 执行当前对象的 `on_event()`
    3. 如果事件在当前层处理后仍未被 `accept()`，则继续沿父对象链向上传播

因此：

    - `send_event()` 适合需要立即观察处理结果的场景
    - `post_event()` 适合解耦调用方与处理时机的场景
    - 二者进入对象后的处理顺序与传播规则完全一致

    `runtime_context` 还提供两类 posted-event 队列管理操作：

    - `send_posted_events(target, type)`：立即冲刷当前已经入队且匹配条件的 posted events。它只影响调用时已经在队列中的事件，不改变之后新入队事件的处理时机。
    - `remove_posted_events(target, type)`：移除当前仍处于 pending 状态且匹配条件的 posted events。已经开始处理的当前事件不受影响。

    posted-event 队列在同一对象内部按优先级和入队顺序处理：

    - 优先级更高的 posted event 先处理
    - 相同优先级下，先入队的 posted event 先处理
    - 同一对象的 posted events 始终串行，不会并发进入同一个对象的事件处理链

    `delete_later()` 也是基于 posted-event 队列实现的延迟销毁操作。调用后，对象不会立刻析构，而是等到该对象事件队列轮到延迟销毁事件时再结束生命周期；对象销毁时，仍处于 pending 状态的 posted events 会一并清理。

    ## #5.2 `on_event()`

`on_event()` 是对象处理事件的主入口。

    当事件到达某个对象时，该对象可以通过重写 `on_event()` 来处理自己关心的事件类型。

    如果对象不关心该事件，可以忽略它，或仅对特定类型进行处理。

    ## #5.3 `event_filter()`

`event_filter()` 和 `install_event_filter()` 用于在 `on_event()` 之前拦截事件。

    它的语义边界应理解为：

    - 过滤器处理的是已经到达当前对象的事件 - 过滤器不负责扩大事件传播范围 -
    过滤器不是广播订阅机制

    因此，安装在某一层对象上的过滤器，并不意味着底层所有事件都必须额外传到这一层。

    ## #5.4 向父对象链传播

    当事件在当前对象处理完成后，如果它仍未被 `accept()`，则该事件会继续传给父对象。

    这一规则不因同步投递或异步投递而改变：

    - 同步投递：当前对象处理完成后，立即继续进入父对象 -
    异步投递：当前层处理完成后，再决定是否继续投递到父对象

    父对象收到事件后，仍然按相同顺序处理：

    1. 本级过滤器 2. 本级 `on_event()`

    因此，`accept()` 的语义，在同步和异步场景下都保持一致：它都会阻止事件继续沿父对象链向上传播。

    异步的父链传播采用逐层推进方式：

    1. 先把事件投递到当前目标对象 2. 等当前对象完成处理后，再根据它是否已被 `accept()` 决定是否继续投递到父对象
    3. 逐层重复这一过程

    这样，同一个事件在某一层被接受或停止传播后，后续父层就不会再收到它，异步传播规则也能与同步传播保持一致。

    因此，对象树为事件提供了结构化的父链传播路径。

    mcbase 不提供面向子对象或子树的通用事件发送机制。事件机制只定义“投递到目标对象”以及“沿父对象链传播”两类语义。

    ## #5.5 简单示例

```cpp class worker : public mc::object {
protected:
    void on_event(mc::event& e) override
    {
        if (e.type() == my_event::type_id) {
            e.accept();
        }
    }
};

worker   obj;
my_event event;
obj.send_event(event);
obj.post_event(std::make_unique<my_event>());
```

    ##6. `signal` 与 `event` 的关系

    ## #6.1 分工不同

`signal` 和 `event` 的分工如下：

    - `signal`：显式连接、同步通知、回调触发 - `event`：对象处理、过滤，以及父对象链传播

        因此：

    - `signal` 不是事件传播机制 - `event` 不是连接管理机制

    ## #6.2 为什么需要两套机制

        保留两套机制的原因是它们面向不同问题：

    - 有些场景需要显式连接和可控的回调顺序 -
    有些场景需要对象自己处理事件，并允许沿父对象链继续传播

        如果把二者强行合并，会导致：

    - `signal` 承担不必要的对象树语义 - `event` 承担不必要的连接管理 -
    显式回调和对象事件处理边界变得模糊

    ## #6.3 如何配合

        在同一个系统中，`signal` 和 `event` 可以同时存在：

    - `signal` 用于本地或显式通知 - `event` 用于对象体系内处理，以及必要时沿父对象链传播 - `mc::
        object` 用于把两者组织到同一个对象模型中

            但 mcbase 不要求“所有 `signal` 必须先转成 `event`”，也不要求“所有 `event` 必须再转成 `signal`”。

    ##7. 何时使用 `signal`

    以下场景更适合使用 `signal`：

    - 需要显式建立连接关系 - 需要同步通知多个回调 - 需要控制 slot 的优先级 -
    不需要依赖父对象链传播

    ##8. 何时使用 `event`

    以下场景更适合使用 `event`：

    - 事件应由对象自身处理 - 需要在对象级做过滤和拦截 -
    需要沿父对象链继续传播

    ##9. 性能边界

        mcbase 需要兼顾嵌入式场景，因此事件机制必须注意以下边界：

    - 不应默认把所有事件无条件沿整条父链广播 - 不应因为安装过滤器而隐式扩大事件传播范围 -
    不应让对象树深度直接线性放大高频路径成本

        因此：

    - 高频通知优先考虑 `signal` 或本级事件处理 -
    事件传播范围应保持明确、可控、可解释

    ##10. 总结

        mcbase 的基础对象通信机制由三部分组成：

    - `mc::object`：对象树、对象级连接、事件处理入口 - `mc::signal`：轻量同步通知 - `mc::
        event`：对象处理、过滤和父链传播 - `runtime_context`：运行时宿主，内部持有 `event_dispatcher`

    这三部分的边界应保持清晰：

    - `signal` 负责通知 - `event` 负责事件对象本身及其处理状态 - `object` 负责对象侧处理入口与对象树结构 - `runtime_context` 内部的 `event_dispatcher` 负责事件投递和传播
